# Refactor Skill System

A set of four Claude Code slash commands for lifecycle-managed refactoring. Works in any repo. A human-authored plan
feeds into a machine-maintained tracking file; agents execute one phase at a time and must pass a validated checkpoint
before the next phase begins.

## Why this exists

Large refactors fail silently. Subagents lose context between sessions, skip Definition of Done criteria, and leave the
codebase in an ambiguous half-migrated state. This system provides a structured handoff protocol: every phase is
model-assigned, dependency-ordered, and checkpoint-gated before it can advance.

## Lifecycle

```text
PLAN.md → REFACTOR.md → execute phases → checkpoint → complete
```

| Stage | Meaning |
| --- | --- |
| PLAN.md | Human-authored plan describing phases, steps, and dependencies |
| REFACTOR.md | Machine-maintained tracking file generated by `/refactor-init` |
| Not started | Phase is eligible or waiting on dependencies |
| In progress | A subagent is actively executing steps |
| Complete | All steps done, checkpoint passed |

## File layout

```text
~/.claude/commands/          <- these skill files (global, works in any repo)
  refactor-init.md
  refactor-next.md
  refactor-checkpoint.md
  refactor-status.md

<repo>/
  .local/
    PLAN.md                  <- human-authored plan (input to /refactor-init)
    REFACTOR.md              <- machine-maintained tracking file
    .gitignore               <- auto-created; excludes .local/ from commits
```

## REFACTOR.md structure

```text
# Refactor: [Title]

## Intent           <- what is being built and why
## Hard Constraints <- must-not-change rules and API contract guards
## Model and Effort Guide
## Definition of Done
## Session Handoff Protocol
## Orchestral Operation
## Test Command
## Phase Map        <- one row per phase: title, status, deps, model
## Session Log      <- append-only history of completed phases

## Phase 1 — [Title]
  Standard Preamble
  Step Index        <- status table for each sub-step
  Step 1.a ...
  Step 1.b ...
  Definition of Done
```

The Phase Map and Step Index are maintained by the skills. Do not edit status values by hand.

## Model assignments

`/refactor-init` assigns a model to each phase during setup:

| Model | Work type |
| --- | --- |
| Haiku | Test-only phases, mechanical lookups |
| Sonnet | Implementation with any judgment, component architecture, test design |
| Opus | Central contracts that cascade into multiple later phases (use sparingly) |
| Codex | Fully-specified mechanical phases with no design decisions left to the agent |

## Commands

### `/refactor-init`

Converts `.local/PLAN.md` into `.local/REFACTOR.md`. Runs on **Opus** — this is the only point where design judgment
is applied to the whole plan.

Before writing anything, Opus reviews the plan for risks, gaps, and ambiguities, then asks clarifying questions. It
assigns a model to each phase, checks for spec completeness, weak Definitions of Done, dependency ordering issues, and
parallel-safety conflicts. Answers are incorporated before the file is written.

```text
/refactor-init
→ Reviews PLAN.md, asks clarifying questions
→ Writes .local/REFACTOR.md with all phases, model assignments, and DoD
→ Outputs: phase summary table + any flagged issues
→ Next: /refactor-status to confirm, then /refactor-next to start
```

### `/refactor-status`

Reports the current state of `.local/REFACTOR.md`. Read-only — nothing is run or changed. Haiku is sufficient.

```text
═══════════════════════════════════════
 Refactor — [Title]
═══════════════════════════════════════

Phase 1 — [name]                        COMPLETE
  ✓ 1.a  [step description]             Sonnet
  ✓ 1.b  [step description]             Sonnet

Phase 2 — [name]                        IN PROGRESS
  ✓ 2.a  [step description]             Sonnet
  → 2.b  [step description]             Sonnet

Phase 3 — [name]                        NOT STARTED — waiting on Phase 2

───────────────────────────────────────
 Next phase:  2 — [name]
 Model:       Sonnet
 Run:         /refactor-next
═══════════════════════════════════════
```

### `/refactor-next`

Runs the next incomplete phase whose dependencies are all complete. One phase per invocation — does not chain.

Finds the eligible phase, constructs a prompt from the Standard Preamble, phase steps, and closing instructions, then
spawns a subagent at the assigned model.

**Claude models (Haiku / Sonnet / Opus):** Spawns a foreground subagent. Reports a synopsis when done and reminds you
to run `/refactor-checkpoint` before the next phase.

**Codex:** Cannot be auto-spawned. Writes the full prompt to `.local/CODEX-PROMPT.md` for the user to paste
into their Codex session.

When re-running a phase that is partially complete (e.g. after a failed checkpoint), the subagent is instructed to
skip steps already marked `Complete` in the Step Index and verify their results instead of redoing them.

```text
/refactor-next
→ Identifies next eligible phase
→ Spawns subagent at assigned model
→ Subagent executes all steps, runs tests, updates REFACTOR.md
→ Reports: what was done, blockers, next phase name + model
→ Next: /refactor-checkpoint before advancing
```

### `/refactor-checkpoint`

Validates the most recently completed phase against its Definition of Done before the refactor advances. Run this
after every phase.

Checks:

- **Tests** — runs the exact command from `## Test Command`; hard stops on failure
- **Code quality** — scans modified files for hacks or untracked workarounds
- **Technical debt** — flags any `// TODO` or `// FIXME` not tracked in the session log
- **Documentation** — checks README, AGENTS.md, doc.go, OpenAPI spec if behavior changed
- **Test coverage** — warns if changed logic has no new or updated tests
- **ADRs** — verifies required ADR files exist in `docs/decisions/`

```text
═══════════════════════════════════════
 Checkpoint — Phase N — [name]
═══════════════════════════════════════
 Tests            PASS
 Code quality     PASS
 Technical debt   PASS
 Documentation    PASS
 Test coverage    WARN — no new tests for changed logic
 ADRs             N/A
 DoD checklist    PASS — 4 of 4 items met
───────────────────────────────────────
 Overall:  PASS — phase is complete
═══════════════════════════════════════
```

On **FAIL**: rolls the phase back — its Phase Map row (and any failed Step Index rows) return to `In progress`, a
`Checkpoint failed` row is appended to the Session Log, and the report lists exactly what must be fixed. The next
`/refactor-next` re-runs the phase, skipping steps still marked `Complete`.

On **WARN**: phase can advance; flags what to revisit.

On **PASS**: confirms Step Index and Phase Map are marked `Complete`, then tells you to run `/refactor-next`.

## Typical session

```text
/refactor-init            <- convert PLAN.md, assign models, ask questions
/refactor-status          <- confirm the generated plan looks right
/refactor-next            <- execute Phase 1
/refactor-checkpoint      <- validate Phase 1 before advancing
/refactor-next            <- execute Phase 2
/refactor-checkpoint      <- validate Phase 2
...
/refactor-status          <- confirm all phases complete
```

## Codex workflow

For phases assigned Codex, `/refactor-next` writes the full prompt to `.local/CODEX-PROMPT.md`. Paste it into your
Codex session. When Codex finishes:

1. Confirm tests pass.
2. Confirm the Step Index and Phase Map rows in `.local/REFACTOR.md` are marked `Complete`.
3. Run `/refactor-checkpoint` to validate the Definition of Done.
